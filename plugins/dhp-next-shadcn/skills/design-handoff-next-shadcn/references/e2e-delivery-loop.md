# 产品到研发的一体化交付链路

当用户给的是功能想法、产品方案、页面草图或半成品协议，而不是完整 DHP 包时，先走这条前置链路，再进入代码实现。

## 1. 先判断当前层级

先用一句话点明当前讨论的是哪一层：

- `产品边界`
- `页面设计`
- `交互规则`
- `实现细节`

不要把多个层级混在同一段里混讲。

## 2. 固定顺序

始终按这个顺序推进：

1. 文档目标
2. 不讨论什么 / 不属于本模块的内容
3. 模块边界 / 一等对象 / 禁止混用的概念
4. 页面职责 / 主入口 / 主动作
5. 信息架构 / 布局分区 / 只读与可编辑边界
6. 操作流程 / 状态机 / 正常态反馈 / 异常态
7. 数据结构 / API 影响
8. V1 / V2 范围切分
9. 测试重点 / 验收标准

## 3. 前置交付物

在协议不完整或需要从 0 到 1 设计时，至少补齐这些产物：

- 产品边界与对象模型
- 页面职责与信息架构
- ASCII 页面线框
- Mermaid 结构图、流程图或状态机
- `dhp-layout`
- `dhp-state-matrix`
- `dhp-binding`
- `dhp-interaction`
- API contract / 数据类型
- V1 / V2 与验收清单

## 4. 强约束

- 不要用 UI 命名反推产品概念。
- 不要把 `状态 / 操作 / 监控 / 配置` 混成一个概念。
- 一页只承担一个主目标。
- 必须明确写出“不属于本模块”的内容。
- 不要只写 happy path，必须补齐禁用态、失败态、恢复策略。
- 缺少边界、对象模型、组件映射、API 契约时，不要 silent fallback 成“先随便做个页面”。

## 5. 进入代码阶段的门槛

满足下面条件后，才能进入前后端实现：

- 主链路对象模型已确定
- 页面主目标与主动作唯一
- 状态机与异常态已定义
- DHP 协议块可落地
- API 字段与前端状态绑定可对齐
