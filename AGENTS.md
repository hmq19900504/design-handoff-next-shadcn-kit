# Agent Instructions

本仓库统一使用 `AGENTS.md` 作为协作代理规范真源。

## 兼容策略

- `Codex`、`Cursor`、`OpenCode`：优先读取本文件。
- `Claude Code`：通过根目录 `CLAUDE.md` 转发到本文件。
- `Copilot`：默认先兼容 `AGENTS.md`；如后续发现某个使用面无法稳定命中，再补极简 `.github/copilot-instructions.md`。

## P0 铁律

以下规则 **每次对话都必须遵守**，违反任何一条即视为方向性错误。

1. 始终使用中文回复。
2. 输出格式：结论先行 -> 最多 3 个关键理由 -> 明确下一步。
3. 交付以可执行结果为主，不只停留在文字建议。
4. 修改前先读上下文；新功能优先复用现有架构，避免重复造轮子。
5. 默认遵循 TDD：先写失败测试，再写最小实现，再重构。修 Bug 先补复现测试。
6. 不要把错误态伪装成空态；失败与无数据必须分开处理。
7. **Fallback 不是默认正确答案。** 会掩盖能力缺口、破坏安全边界、污染数据语义、或把失败伪装成成功时，禁止 silent fallback，必须显式报错或 fail-closed。详细分级见 `agent-rules.md#fallback-分级规则`。
8. 产品内模型负责高不确定性语义判断；代码负责约束、执行和 fallback。不要用硬规则替代模型应做的意图理解。
9. 同主题优先更新现有文档，不重复造新文档。规则只更新 `AGENTS.md` 或共享细则。
10. **不要频繁把未完成的问题回抛给用户。** 只要还存在明显主路径缺陷、未验证项、可继续独立推进的实现空洞，当前协作代理就应继续自行解决；不要为了交差引入旁门左道的 case、临时特判、silent fallback 或把缺陷藏起来。只有在以下情况才应追问用户：产品边界缺失且会显著影响方案；存在高风险/不可逆操作；缺少必要权限、外部资源或真实数据；多个方向都合理但会实质改变交付。默认由当前协作代理先按专业标准自检完成度至少达到 `8/10` 再决定是否提问；这里的 `8/10` 指主路径可用、关键测试通过、已知风险已显式说明、失败态未被伪装。

## 术语边界

- `当前协作代理`：当前会话里帮你改仓库的编码代理（Claude / Codex / Cursor）。
- `产品内 Agent`：Manuspro 运行时中的 AI 模块，是代码里的业务对象，不是当前协作代理。
- 对当前协作代理用"你 / 请"；对产品内模块用"产品内… / 运行时… / 模块…"。

### 权重标记含义

- 🔴 **MUST** — 违反即事故或数据损坏，必须无条件遵守
- 🟡 **SHOULD** — 高频踩坑经验，强烈建议遵守，跳过需说明理由
- 🟢 **NICE** — 最佳实践，有余力时遵守

## 沟通边界

- 追问产品边界时，先点明当前层级再展开。
- 讨论产品方案时，先定义对象模型，再谈页面。
- 区分"事实状态""可执行动作""观测指标""运行策略"。
- 是否提问先看自己是否还能继续独立推进；能通过读代码、跑测试、补验证、查现有文档解决的，就不要先问用户。
- 判断"是否做完"时用专业标准，不用"能不能先交差"标准：主链路未通、关键验证未过、明显缺陷还在、失败被 fallback 掩盖，这些都不算完成。

## 产品设计审美原则

以下原则适用于 Inspector、详情面板、配置面板、侧栏等信息展示型 UI 的设计评审和方案输出。

### 信息标签化

- 🔴 关键属性用标签（badge / tag / chip）压缩展示，不用大段文字描述。状态、类型、来源系统、同步方式等都应该是**一眼可扫的标签**，不是需要阅读的段落。
- 🔴 字段列表、属性列表等结构化信息直接平铺展示，用紧凑布局（字段名 + 类型标签），不要默认折叠再让用户手动展开。折叠适用于低频且体量大的信息（如变更历史），不适用于用户经常需要看的内容。

### 流线型信息流

- 🔴 信息从上到下有一条自然主线：身份 → 关键属性 → 数据概览 → 可执行动作。用视觉密度和间距自然过渡，不要用大量分割线把信息切成孤立区块。
- 🟡 同一面板内的信息消费应该是"扫一遍就知道全貌"，不是"打开 → 折叠 → 展开 → 再折叠"的来回操作。

### 就地消费，减少跳转

- 🟡 能在当前面板内完成的信息消费就不弹新窗口、不跳新页面。Popover、inline 展开等轻量交互优先。
- 🟡 跨模块导航（如"查看接入配置"）可以跳转，但必须视觉降级（文字链接，不是按钮），并且和本模块内的主动作明确分开。

### 术语零发明

- 🔴 不生造新术语、新文案。上位文档（`04-04` 等）里叫什么，面板里就叫什么。如果上位文档叫"新建 Fact"，面板里就是"新建 Fact"，不要改成"基于此表创建模型"。
- 🔴 如果发现现有术语有歧义或不准确，正确做法是**先改上位文档的术语定义**，再同步到 UI，而不是在 UI 层单独造一个新说法。

### 苏格拉底式反问

设计方案前和输出方案后，必须对自己的每个设计决策做以下反问，并在文档中显式回答至少前三条：

1. **这个信息用户真的需要在这里看到吗？** — 如果删掉它，用户会损失什么？
2. **这个操作用户在这个上下文里真的会做吗？** — 有没有更自然的触发时机？
3. **有没有更简单的方式达到同样目的？** — 是否引入了不必要的层级、弹窗或中间步骤？
4. 这个设计解决的是用户的问题，还是设计者自己的结构洁癖？
5. 如果把这个方案给一个从没见过产品的人看，他能在 5 秒内理解面板在干什么吗？
